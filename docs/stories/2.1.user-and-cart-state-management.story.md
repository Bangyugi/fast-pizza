# Story 2.1: User and Cart State Management

## Status

Draft

## Story

**As a** Developer,
**I want** to set up centralized state management for user and cart data using Redux,
**so that** this data is consistently handled and accessible throughout the application.

## Acceptance Criteria

1.  Redux Toolkit (`@reduxjs/toolkit` and `react-redux`) is integrated into the project.
2.  A `userSlice` is created to store the customer's name. On the Home page, an input for the name is captured and stored in Redux.
3.  A `cartSlice` is created with actions to add items, remove items, adjust quantities, and clear the cart.
4.  The Redux store is correctly configured and provided to the React component tree.

## Dev Notes

### Project Structure Notes

Based on the project's established feature-based structure, new Redux slice files should be co-located with their features. The existing file `src/utils/userSlice.js` is misnamed and contains geolocation logic; it is **NOT** a Redux slice and should be ignored for this task. The new store configuration file should be created at `src/store.js`.
[Source: `docs/brownfield-architecture/source-tree-and-module-organization.md`]

### Previous Story Insights

Story 1.4 was purely for rendering UI and explicitly deferred all state and action handling. The project currently has no centralized state management.

### Tech Stack

The project uses React and Vite. This story introduces Redux Toolkit for state management.
[Source: `docs/brownfield-architecture/high-level-architecture.md`]

### Data Models

This story will define the client-side data models for the `user` and `cart` within the Redux slices. The `cartSlice` should be prepared to hold pizza objects, whose structure is implicitly defined by the API. A typical cart item should include at least `pizzaId`, `name`, `quantity`, and `totalPrice`.
[Source: `docs/brownfield-architecture/data-models-and-apis.md`]

### API Specifications

This story does not directly involve API calls, but it sets up the state management that will be used for API interactions in subsequent stories (e.g., placing an order).
